# Roadmap v3 — 讓系統活過來(2026-07-10,由 Claude 規劃、Nelson 授權)

> v2 收斂時的系統:完整、自我防衛、公開可驗證 —— 但**靜止**。
> 14 個 claim 是快照;來源被引用之後,系統就不再看它們一眼。
> 真實世界不是這樣運作的:論文被更正、被加編輯關注、被撤稿。
> **v3 的主軸:知識會動,來源也會動 —— 兩者都不准無聲地動。**

## 現實給的活案例(v3 的試金石)

`dark_oxygen_production` 引用的 Sweetman 2024(Nature Geoscience):
2024 年多篇反駁 → 作者收回部分主張 → **2026 年 4 月期刊加註編輯關注**。
我們引用它時誠實記錄了爭議,但系統**沒有任何機制**在期刊動作時察覺。
T1 就是為此而生,而且上線第一天就有真資料可咬。

## v3 項目(依價值排序)

### T1 來源健康檢查(旗艦)— Retraction/Update Watch

- 機制:Crossref 的 `filter=updates:<DOI>` 反查 —— 更正、勘誤、撤稿都是
  「一篇 update-to 指向原文的新 DOI」。對每個已引用 DOI 定期反查。
- 產出:健康報告(機械記錄:更新方 DOI、類型、日期,**零詮釋**);
  新發現 → 事件檔 → **走既有管線進 feed**(訂閱者自動收到);
  排程 CI(每週)自動跑,有新發現就開 GitHub issue 提醒人類重審。
- 治理邊界(誠實劃線):健康檢查**只報告,不判定**。來源被撤稿後 claim
  該怎麼辦,是人的重審;「引用已撤稿來源卻無 status_history 回應」是否
  該入憲,登記為 **R9**,留給有真實案例時修憲。
- arXiv 撤回偵測留 T1+(arXiv API 對 withdrawn 的表達較弱)。
- **實測記錄(2026-07-10)**:9 筆引用 DOI 全 clean —— 包括 Sweetman 2024。
  機制本身經 Wakefield 1998 驗證有效(正確抓到 2004 部分撤回 + 2010 正式撤稿),
  結論:Nature Geoscience 的編輯關注**尚未以正式 update 記錄存繳 Crossref**。
  T1 的誠實邊界 = 只抓正式存繳的更正/勘誤/撤稿;期刊一正式動作,週期檢查即響。
  補充訊號源(Retraction Watch 資料庫,已併入 Crossref)登記為 T1+。

### T2 挑戰管道 — 让「可被推翻」有真實入口

- GitHub issue 模板:「推翻一個判定」(引導填:claim id、你主張違反哪條
  入格條件、你的依據來源)+「回報來源問題」。
- CONTRIBUTING.md:第三方如何覆核(claims.json → §3 條件 → 提出挑戰)。
- 網站 claim 卡片加「challenge this」連結直達 issue 模板。
- 價值:系統的核心賣點是「可被推翻」,但目前沒有陌生人能走的推翻路徑。

### T3 憲章頁 — 讓陌生訪客三分鐘看懂

- 網站加 about.html(雙語):五格燈號怎麼讀、證據軸怎麼湧現、
  什麼是分岔、如何挑戰一個判定。內容從既有文件濃縮,不新造宣稱。

### T4 LLM 輔助 claim 起草管線(B3)

- 現有接縫的集大成:candidates 搜尋 → 本地 LLM 起草 claim 骨架(含
  status_reason 草稿)→ **全部既有把關照跑** → 人審核後才入 data。
- 這是內容規模化的引擎;沒有它,T5 就是純手工。

### T5 內容擴張

- 目標:每領域 8+ claims 或第四領域(地震學/深時地質)。由 T4 加速。

## 不做(v3 明確劃線)

- ❌ 不做使用者帳號/留言系統 —— 挑戰走 GitHub,身分與紀錄免費得到。
- ❌ 不做「撤稿自動降燈」—— 自動判定仍是禁區,健康檢查只餵給人。
- ❌ 不做全文抓取(版權雷區)—— metadata 層足以支撐現有憲法。

## 執行順序

T1(本回合開工)→ T2(下一手,便宜)→ T3 → T4 → T5。
