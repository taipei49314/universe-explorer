# P2 Spec — 半自動判燈(機械提案 → 人拍板 → 稽核軌跡)

> 定位:Roadmap P2,治理支柱的升級。走 A 路線(機械提案引擎,零 LLM 依賴):
> 先把「提案 → 人審 → 稽核」的治理管線跑通;未來若換 LLM 提案器,把關結構不變。

---

## 0. 治理鐵律(本版憲法延伸)

1. **提案不是判定。** 引擎輸出的是「哪些燈號與機械訊號相容 + 人還需要確認什麼」,
   永遠不寫入 claim 資料。不存在任何程式路徑讓提案自動落地。
2. **機械訊號誠實分界。** 每個入格條件標記為:
   - `machine`:可從已收錄證據機械判讀(如「多重獨立直接觀測」← 證據軸 E1)
   - `human`:機器**不可能**判讀(教科書接受度、學界是否真有兩派、peer review 狀態)
   引擎對 human 條件一律回答「無法判讀,需人確認」,**絕不猜**。
3. **機械訊號可以排除、不能核准。** 一個燈號被排除(如證據軸 E3 ⇒ Established 的
   「多重獨立驗證」條件不成立)是機械事實;但「該給哪個燈」永遠是人的決定。
4. **每個決定留稽核。** 人審提案(接受/駁回)寫入 append-only 稽核日誌:
   時間、claim、提案內容、決定、決定者、理由。可回溯、可翻案。

## 1. 範圍

### 做
- `proposals.py`:條件判讀器(machine/human 分界表)+ `propose(claim)` +
  審核 CLI(`--decide`,寫 JSONL 稽核日誌)。
- 對真實 4 claim 跑提案,做一致性檢查:人已判的燈號必須在引擎的相容集合裡
  (若不在,表示要嘛資料錯、要嘛引擎錯 —— 這是好事,系統抓得到)。
- 測試:提案可回溯、排除邏輯正確、human 條件絕不被機器猜、稽核日誌可寫可讀。

### 不做
- ❌ LLM 提案(B 路線,留待把關管線證明穩之後)。
- ❌ 提案自動落地(永遠不做,這是鐵律不是階段)。
- ❌ 燈號自動遷移(P3 只偵測與擋關,遷移仍人工)。

## 2. P2 驗收(先於實作)

1. **提案附證據**:提案中每個 machine 訊號都引用具體證據記錄/推導規則;
   human 條件明確標示「機器無法判讀」。
2. **提案不落地**:程式碼中不存在從 proposals 寫入 data/ 的路徑;測試驗證
   審核 CLI 只寫稽核日誌。
3. **一致性成立**:4 個真實 claim,人工判的燈號都落在引擎相容集合內;
   且引擎能**排除**至少一個不相容燈號(證明它不是照單全收)。
4. **稽核可回溯**:決定記錄含提案快照 + 決定者 + 理由;append-only。
5. **零違憲不退步**:全部既有測試照過。
