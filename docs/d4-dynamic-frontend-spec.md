# D4 Spec — 動態前端(Nelson 提議,2026-07-10)

> v0 的「前端刻意陽春」是為了先證明認識論;證明完了。
> D4 的原則繼承 design-framework:**視覺服務於「看見知識形狀」,不是特效。**
> 每一個動態元素都必須回答:它讓知識的形狀更可見了嗎?

## 0. 架構決定

- **靜態託管不變、$0 不變**:動態 = 客戶端(vanilla JS)讀 `app-data.json`
  (build 時從引擎資料生成,含中英雙語 overlay)。無框架、無 build step、
  無 CDN —— 單檔自足,跟系統其他部分同一性格。
- **陽春版保留**:既有頁面(index/topic/explore)是可及性底線與 permalink
  載體,動態 app 是**另一個視圖**(`app.html`),不取代、不破壞任何連結。
- 憲法照管:無信心數字;計數皆為可見清單之計數;敘事逐句帶 refs 照顯示。

## 1. 核心視圖:雙軸知識地圖(本版的靈魂)

系統最深的結構 —— **共識軸 × 證據軸** —— 至今只以徽章呈現。D4 把它變成
真正的二維地圖:

- X 軸:共識(Established → Speculative),Y 軸:證據(E1 → E5)。
- 每個 claim 是一顆按燈號著色的點;**⚡ 分岔 claim 天然落在對角線之外** ——
  分岔第一次成為肉眼可見的空間事實,不是徽章。
- 同格多點自動錯開;點擊 → 詳情面板;領域篩選 chips;懸停顯示標題。

## 2. 其他動態元素(全部服務形狀)

- **詳情面板**:點擊地圖或清單開側欄 —— 完整 claim(逐條 status_reason、
  證據含分級章、競爭模型、開放問題、AI 敘事、出處)+ permalink +
  challenge 連結。
- **即時搜尋/篩選**(沿用 explore 的邏輯,套進地圖與清單雙視圖)。
- **語言切換**(en/中文)純客戶端,吃同一份 app-data。
- 動效節制:面板滑入、點的 hover 放大;`prefers-reduced-motion` 全域尊重。
- 鍵盤可達:地圖點可 tab、Enter 開面板、Esc 關。

## 3. 驗收

1. `app-data.json` 完整一致:全部 claim、雙語欄位、軸值與引擎一致(測試)。
2. app.html 自足:無任何外部資源請求(測試);單檔可離線開啟。
3. 分岔在地圖上肉眼可辨(偏離對角線),與 `diverges` 機械判定一致。
4. 陽春版每個 permalink 照常;index 增「Dynamic view」入口。
5. 全部套件照過;CI 部署後線上可用。
