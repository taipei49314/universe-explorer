# P3 Spec — 變化偵測與事件(從「快照」到「活的」知識)

> 定位:Roadmap P3。status_history 不再只是被動欄位:系統偵測知識狀態的變化、
> 產出可回指證據的事件(未來推送的基礎),並機械強制「燈號遷移必須留下歷史」。

---

## 0. 核心機制

1. **狀態快照**(`snapshot/state.json`):每個 claim 記錄
   `status`、`evidence_axis`(推導值)、`evidence 條數`、`sources 條數`、`diverges`。
2. **diff → 事件**:當前狀態 vs 快照有差異 → 產出事件檔(`events/*.json`)。
   每個事件必須**可回指證據**:寫明哪個 claim、什麼變了、變化前後、以及當下的
   推導理由(不是 AI 的詮釋,是機械 diff)。
3. **未記錄遷移 = 違憲**(本版最重要的一條新憲法):
   claim 的 `status` 與快照不同,但 `status_history` 沒有一筆
   `to_status == 新狀態` 的記錄 → `undocumented_status_change`,build 擋下。
   燈號可以變,但**不准無聲地變** —— 每次遷移都要有日期與觸發原因。
4. **快照更新是顯式動作**(`--commit`):看過 diff、確認事件無誤,才把新狀態
   寫入快照。跟 P2 一樣:偵測是機器的,確認是人的。

## 1. 範圍

### 做
- `watch.py`:快照、diff、事件產出、`undocumented_status_change` 檢查、`--commit`。
- build.py 把「未記錄遷移」檢查納入擋關(快照存在時)。
- 測試:偽造變化 → 事件正確產出;無聲改燈 → 抓到;有寫 history → 放行。
- 建立初始快照。

### 不做
- ❌ 不做真實推送管道(email/webhook)—— 事件檔就是推送的觸發介面,管道是 P5。
- ❌ 不做自動改燈(遷移永遠由人寫進 data + status_history,機器只負責抓「沒寫」)。
- ❌ 不做排程輪詢(那是部署問題,不是模型問題)。

## 2. P3 驗收(先於實作)

1. **遷移有觸發條件、可回溯**:每筆 status_history 都有 date/from/to/trigger;
   無聲改燈被機械擋下(測試證明)。
2. **事件可回指證據**:事件檔含變化前後值 + 當下推導理由,不含任何詮釋性文字。
3. **穩定時零噪音**:資料未變時,watch 不產出任何事件。
4. **快照更新顯式**:不 `--commit` 就不動快照;連跑兩次 diff 結果一致(冪等)。
5. **零違憲不退步**:全部既有測試照過。
