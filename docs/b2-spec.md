# B2 Spec — 本地 LLM 提案器(R2 的 B 路線落地)

> P2 建好的治理閘:提案不落地、機械只排除不核准、human 條件絕不猜、稽核 append-only。
> B2 讓本地 LLM 進場,但它的活動範圍被閘死在一個格子裡。

## 0. LLM 在判燈治理裡的唯一職權

機械引擎已經把條件分成兩類:machine(從證據推導)與 human(教科書接受度、
學界是否真有兩派⋯)。**LLM 只准對 `needs_human` 條件起草評估建議** ——

- ❌ 不准碰 machine 條件(那是推導,不是意見)。
- ❌ 不准提議燈號(相容集合是機械算的)。
- ❌ 草稿不落地:寫入 `audit/drafts.jsonl`,每筆標記 `UNVERIFIED`,
   與人類決定日誌(decisions.jsonl)物理分離。
- ✅ 只產出:「條件 X:建議 holds=true/false + 草擬理由」,供人拿真實來源覆核。

草稿的認識論地位 = **待驗證的假設**,不是判定、不是證據、不是理由。
人若採用,要自己把理由(附真實依據)寫進 data 檔 —— 然後 v0 憲法照常接管。

## 1. 把關(全部沿用,零新法院)

- 條件鍵合法性:草稿裡的條件必須是該燈號的 human 條件(機械檢查)。
- 文字法:宣告信心語彙殺、裸 % 殺(validator 同一套正則)。
- 失敗即退回:模型沒開/格式壞/違規 → 只回機械提案,無草稿。

## 2. 驗收

1. 越權草稿被殺:偽造 LLM 輸出碰 machine 條件 / 不存在的條件 → 拒收(測試)。
2. 文字法平等:草稿含「共識度 90%」→ 拒收(測試)。
3. 物理分離:drafts.jsonl 與 decisions.jsonl 不同檔;草稿全帶 UNVERIFIED 標記(測試)。
4. 離線安全:模型沒開時 CLI 照常輸出機械提案(測試)。
5. 活體驗證:真實 claim 上本地模型產出合法草稿,或誠實記錄被拒。
