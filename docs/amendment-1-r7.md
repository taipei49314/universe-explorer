# 憲法修正案 #1 — 實測數字與宣告信心的區分(銷 R7)

> 定位:第一次修改 validator(憲法法院的法條)。P4 實測發現 `no_fake_precision`
> 誤殺論文裡的真實實測值(AMOC「~15%」、CCZ「~92%」)。
> 修法有修法的紀律:條文先行、新舊規則都要有測試證明會咬人、引擎基準重新蓋章。

---

## 0. 立法原意回顧(v0 憲法第 3 條)

禁止的是「**AI 主觀捏造的連續數字**」——「共識度 73%」這種宣告出來的信心。
從未禁止「**來源裡真實存在的實測量**」——論文自己說的 15% 減弱。
v0 的裸 `%` 正則是立法原意的過寬實作;修正案讓法條回到原意。

## 1. 修正條文

### 區分兩種數字

| 種類 | 定義 | 處置 |
|---|---|---|
| **宣告信心** | 「共識度/信心/confidence/consensus + 數字」這類把確定性說成數字的語彙 | **全域禁止,包括證據欄**(新規則 `declared_confidence`) |
| **實測量** | 出現在證據描述裡的百分比(而證據描述被 v0 憲法強制掛 source_ref) | **允許** —— 它是來源內容的轉述,不是系統宣告 |

### 機械落地

- `%` 檢查的適用範圍從「所有文字欄位」縮小為「**證據描述以外**的所有文字欄位」
  (title、open_questions、status_reason、competing_models 照禁 ——
  編輯紀律:**量值住在 Evidence 層、掛著出處**;別處要用數字,就把它收錄成證據)。
- 信心語彙檢查(`declared_confidence`)**全域適用**,證據欄也不例外:
  就算掛了出處,也不准把出處改寫成「共識度 73%」這種句式。
- 為什麼安全:證據描述本來就被 v0 憲法強制 `source_ref` 且 ref 必須解析到真實
  Source(P1 又對 arXiv 來源強制 fetch)。百分比因此永遠躺在一條掛著真實出處的
  記錄裡,可被覆核。

## 2. 修法紀律

1. 修正案條文先寫(本文件)。
2. validator 修改後,新舊規則都要有測試:
   - 舊:非證據欄位的裸 `%` 照抓(既有測試不得刪)。
   - 新:證據欄的實測 `%` 放行;證據欄的宣告信心語彙照抓。
3. 海洋資料恢復論文原始數值(~15%、~92%)—— 修法的直接受益者,也是驗收證據。
4. `engine_hashes.json` 基準重新蓋章:P4 的「零改動」測試守的是**未經修正案的暗改**;
   經修正案的修改,蓋新章后繼續守。

## 3. 驗收

1. 海洋證據欄寫回「~15%」「~92%」,validator 放行,build 零違憲。
2. 「共識度 73%」放進任何欄位(含證據欄)都被抓(測試證明)。
3. 非證據欄位(open_questions/status_reason/title)的裸 `%` 照抓(既有測試全過)。
4. 全部測試套件照過;引擎基準重新蓋章後 P4 測試照過。
